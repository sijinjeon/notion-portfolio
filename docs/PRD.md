# **제품 요구사항 정의서(PRD): Notion 기반 개인 웹페이지 자동화 솔루션**

| **문서 버전** | **작성일** | **작성자** | **상태** |
| :--- | :--- | :--- | :--- |
| 1.4 | 2025.10.17 | Gemini | 최종본(Final) |

## **1. 개요 (Introduction)**

콘텐츠 크리에이터, 개발자, 디자이너 등 자신의 작업물을 온라인에 게시하고자 하는 개인들은 포트폴리오나 블로그 웹사이트가 필요합니다. 하지만 별도의 웹 에디터나 복잡한 CMS(콘텐츠 관리 시스템)를 배우고, 콘텐츠를 이중으로 관리하는 것은 비효율적이며 번거로운 과정입니다.

본 문서는 **Notion을 CMS로 활용하여 콘텐츠 작성 시 개인 웹페이지에 자동으로 게시되는 자동화 솔루션**에 대한 제품 요구사항을 정의합니다. 사용자는 이미 익숙한 Notion 환경에서 글을 쓰고 관리하는 것만으로도 전문적인 웹사이트를 운영할 수 있게 하여, 콘텐츠 관리에 드는 시간과 노력을 획기적으로 줄이는 것을 목표로 합니다.

## **2. 문제 정의 (Problem Statement)**

-   **높은 진입 장벽:** 웹사이트를 구축하고 운영하기 위해서는 별도의 개발 지식이나 WordPress, Ghost 등 새로운 툴 학습이 필요하다.
-   **비효율적인 워크플로우:** 콘텐츠 초안 작성(e.g., Notion)과 실제 웹사이트 발행(e.g., CMS 에디터)이 분리되어 있어 반복적인 '복사-붙여넣기' 작업과 이중 관리가 발생한다.
-   **유지보수의 어려움:** 콘텐츠의 사소한 오타나 이미지 변경, 공통 정보(Footer 등) 수정을 위해 여러 단계를 거쳐야 하며, 이는 신속한 업데이트를 저해한다.

## **3. 제품 목표 및 비전 (Goals & Vision)**

-   **목표:**
    -   사용자가 Notion 외에 별도의 에디터 없이 콘텐츠를 웹에 발행할 수 있는 가장 단순한 워크플로우를 제공한다.
    -   10분 이내의 준실시간 동기화를 통해 콘텐츠의 최신성을 보장한다.
    -   SEO에 필요한 정보(Slug, Meta Description 등)까지 Notion 내에서 완결적으로 관리할 수 있도록 한다.
-   **비전:** "콘텐츠는 Notion에, 나머지는 자동으로" — 누구나 자신의 지식과 경험을 손쉽게 온라인 자산으로 만들 수 있는 생태계를 구축한다.

## **4. 타겟 고객 (Target Audience)**

-   **주요 타겟:**
    -   **포트폴리오 아카이빙이 필요한 전문가:** 자신의 프로젝트, 경력, 인사이트를 정리하고 외부에 공유하려는 개발자, 디자이너, 기획자.
    -   **기술 블로거 및 콘텐츠 크리에이터:** 복잡한 설정 없이 글쓰기에만 집중하고 싶은 작가 및 크리에이터.
-   **고객의 니즈:**
    -   쉽고 빠른 콘텐츠 발행을 원한다.
    -   이미 사용 중인 Notion으로 모든 것을 관리하고 싶어 한다.
    -   마크다운, 코드 블록 등 기술 콘텐츠 작성에 용이한 환경을 선호한다.

## **5. 사용자 페르소나 및 시나리오 (User Persona & Scenario)**

### **5.1. 페르소나**

-   **이름:** 김민준 (UX/UI 디자이너, 5년 차)
-   **목표:** 잠재적 고용주와 클라이언트에게 자신의 최신 프로젝트를 효과적으로 보여줄 수 있는 온라인 포트폴리오를 운영하고 싶다.
-   **어려움:** "저는 디자이너이지 개발자가 아니에요. 웹 서버를 세팅하거나 Git 같은 복잡한 툴을 다루는 건 너무 어렵고 시간도 많이 뺏겨요. 그냥 제가 만든 작업물을 쉽고 빠르게 올리고 싶을 뿐이에요."
-   **기술 역량:** Figma, Sketch 등 디자인 툴에 능숙하며, 프로젝트 관리 및 문서화 도구로 Notion을 매우 활발하게 사용 중. 코딩은 피하고 싶어 함.

### **5.2. 행동 시나리오: 새로운 프로젝트를 포트폴리오에 추가하기**

1.  **콘텐츠 작성 (Notion에서):** 민준 씨는 자신의 Notion에 있는 '포트폴리오' 데이터베이스를 엽니다. '새로 만들기'를 눌러 최근에 완료한 '모바일 뱅킹 앱 리디자인' 프로젝트를 위한 새 페이지를 생성합니다.
2.  **세부 정보 입력 (Notion에서):**
    -   `제목` 속성에 '모바일 뱅킹 앱 리디자인'이라고 입력합니다.
    -   페이지 본문에 프로젝트의 목표, 과정, 결과물(이미지, Figma 임베드 등)을 포함한 상세한 케이스 스터디를 작성합니다.
    -   `카테고리` 속성을 'UX/UI Design'으로 선택합니다.
    -   포트폴리오 메인 화면에 보일 `요약` 속성에 한 문장으로 프로젝트를 요약합니다.
    -   `상태` 속성을 '초안(Draft)'에서 **'게시(Published)'**로 변경합니다.
3.  **자동 동기화 (시스템):** 민준 씨가 커피를 마시러 간 사이, 시스템은 10분 주기로 예정된 동기화 작업을 자동으로 실행합니다. (GitHub Actions)
4.  **변경 감지 및 배포 (시스템):** 시스템은 Notion DB에서 '게시' 상태로 변경된 새 페이지를 감지합니다. 해당 페이지의 모든 콘텐츠와 속성 정보를 가져와 웹사이트를 새롭게 빌드하고, 그 결과를 GitHub 저장소에 자동으로 반영(Commit & Push)합니다.
5.  **게시 완료 및 확인 (사용자):** 잠시 후, 민준 씨는 자신의 포트폴리오 웹사이트(`minjun.design`)에 접속합니다. 방금 Notion에서 작업한 '모바일 뱅킹 앱 리디자인' 프로젝트가 목록 최상단에 카드 형태로 추가된 것을 확인합니다. 카드의 'Read More' 버튼을 클릭하자, 자신이 Notion 페이지 본문에 작성했던 상세한 케이스 스터디 내용이 모두 포함된 개별 프로젝트 페이지로 이동합니다. 모든 페이지 하단의 Footer 정보도 정상적으로 표시되는 것을 보고 만족합니다.

## **6. 핵심 기능 요구사항 (Features & Requirements)**

| Feature ID | 기능명 | 설명 | 우선순위 |
| :--- | :--- | :--- | :--- |
| FEAT-001 | **Notion DB 기반 콘텐츠 관리** | Notion 데이터베이스를 웹사이트의 CMS로 직접 연동. DB의 각 페이지가 웹페이지의 게시물이 됨. | **Must-Have** |
| FEAT-002 | **자동 동기화 및 배포** | 지정된 시간(e.g., 10분)마다 Notion DB의 변경사항을 감지하여 웹사이트에 자동으로 빌드 및 배포. (GitHub Actions 등 활용) | **Must-Have** |
| FEAT-003 | **Notion 내 SEO 및 메타데이터 관리** | Notion DB의 속성(Property)을 활용하여 각 게시물의 URL Slug, Meta Title, Description, Thumbnail Image 등을 직접 관리. | **Must-Have** |
| FEAT-004 | **게시물 상태 및 카테고리 관리** | Notion DB의 '상태(Status)'나 '체크박스' 속성을 이용해 게시물의 공개/비공개 상태를 제어하고, 'Select' 속성으로 카테고리를 분류. | **Must-Have** |
| FEAT-005 | **목록-상세 페이지 연동** | 메인 페이지의 각 프로젝트 요약 카드에는 'Read More' 버튼이나 제목 자체에 링크가 포함되어야 한다. 이 링크를 클릭하면 해당 프로젝트의 전체 내용을 담은 개별 상세 페이지로 이동해야 한다. | **Must-Have** |
| FEAT-006 | **공통 컴포넌트 관리 (Footer)** | Notion의 특정 페이지나 별도 DB를 통해 웹사이트의 Footer 정보를 관리할 수 있어야 한다. 여기서 수정된 내용은 메인 페이지와 모든 서브 페이지에 동일하게 동기화되어야 한다. (예: 저작권 정보, 소셜 미디어 링크 등) | **High** |
| FEAT-007 | **템플릿 기반의 손쉬운 커스터마이징** | 사용자가 기본으로 제공되는 웹사이트 템플릿에서 시작하여, 약간의 코드 수정이나 설정 변경만으로 자신만의 디자인을 적용할 수 있도록 지원. | **High** |
| FEAT-008 | **안정적인 서비스 운영** | 초기 설정 이후 사용자의 개입 없이 안정적으로 동기화 및 배포가 이루어져야 하며, 오류 발생 시 추적이 용이해야 함. | **High** |

### **6.1. 벤치마킹 사례 분석: [hong-minji.github.io/works/](https://hong-minji.github.io/works/)**

해당 포트폴리오 사이트는 우리가 만들고자 하는 제품의 좋은 실제 사용 사례를 보여줍니다. 이 구조를 Notion 데이터베이스로 어떻게 구현할 수 있을지 분석합니다.

-   **페이지 구성:** 메인(아카이빙), 소개, 상세 페이지로 구성.
-   **콘텐츠 구조:** 프로젝트 카드는 `카테고리`, `제목`, `요약`, `'Read More' 링크`로 구성.
-   **결론:** 벤치마킹 사이트의 구조는 본 제품의 핵심 기능(FEAT-001, FEAT-004, FEAT-005)이 왜 필수적인지를 명확히 보여줍니다. 사용자는 Notion DB 속성 관리만으로 웹사이트의 카테고리, 요약글, 공개 여부, 상세 페이지 연결을 손쉽게 제어할 수 있어야 합니다.

## **7. 기술적 고려사항 (Technical Considerations)**

-   **프레임워크:** Next.js, Astro, Eleventy 등 정적 사이트 생성(SSG)에 유리한 프레임워크 사용을 권장.
-   **API 연동:** Notion API를 통해 데이터베이스 정보 및 특정 페이지의 블록 데이터를 안정적으로 호출.
-   **자동화 파이프라인:** GitHub Actions 또는 유사 CI/CD 도구를 통해 스케줄 기반의 자동 빌드 및 배포 파이프라인 구축.
-   **호스팅:** Vercel, Netlify, GitHub Pages 등 정적 사이트 호스팅에 최적화된 플랫폼 활용.
-   **컴포넌트 관리:** Footer와 같이 반복 사용되는 컴포넌트의 콘텐츠는 빌드 시점에 특정 Notion 페이지 ID에서 데이터를 가져와 모든 페이지에 일괄적으로 주입하는 방식으로 구현.

## **8. 성공 지표 (Success Metrics)**

-   **동기화 성공률:** 전체 동기화 시도 횟수 대비 성공 비율 (목표: 99.9%)
-   **콘텐츠 발행 시간:** Notion에서 'Published' 상태로 변경 후 실제 웹페이지에 반영되기까지 걸리는 시간 (목표: 평균 10분 이내)
-   **사용자 만족도:** 설정의 용이성 및 운영 안정성에 대한 사용자 만족도 점수 (CSAT)

## **9. 향후 계획 및 비전 (Future Scope / Out of Scope)**

-   **향후 계획 (Future Scope):**
    -   실시간 동기화(Webhooks 기반) 옵션 제공
    -   다양한 디자인의 공식 템플릿 추가
    -   코딩 없이 설정을 완료할 수 있는 GUI 설정 페이지 개발
-   **범위 외 (Out of Scope):**
    -   본 버전에서는 Notion 내 실시간 공동 편집 기능의 동기화는 지원하지 않음.
    -   다국어 지원은 고려하지 않음.
    -   웹사이트 내 댓글, 방명록 등 동적인 기능은 포함하지 않음.
